# 背景
当前成就与学习进度链路依赖 `learning_log` 与 `user_achievements` 两个集合，但仓库中尚未提供对应 schema 文件，导致字段约束、权限边界、唯一性约束和索引要求都未被显式管理。  
现有云函数已经暴露出两类问题：一是按 `user_id` / `(user_id, card_id)` / `(user_id, achievement_id)` 查询和写入频繁，但缺少结构化索引声明与数据契约；二是 `recordLearning`、`checkAchievements` 采用“先查后写”的流程，在并发或重试场景下存在重复学习记录、重复成就、重复积分的风险。  
用户希望先把表设计收口，再继续做查询性能优化，因此本次改动聚焦“把数据模型定义清楚、把约束补齐、让后续性能优化建立在稳定契约上”。

# 目标
- 为 `learning_log`、`user_achievements` 补齐 schema，明确字段、权限、时间字段和对外约束。
- 明确学习记录与成就记录的一致性规则，确保后续实现能围绕“每个用户-卡片最多一条学习记录”“每个用户-成就最多一条成就记录”收敛。
- 在不改变前端接口形态的前提下，为后续索引补齐和性能优化提供稳定的数据模型基础。

# 非目标
- 本次不直接重写 `getLearningProgress` 的慢查询实现，不把“去掉 N+1 查询”与 schema 改造混在同一轮完成。
- 本次不改页面展示、接口返回结构和业务文案。
- 本次不做破坏性数据迁移；若现有线上数据已存在重复记录，仅定义约束和兼容处理边界，不在本轮引入全量清洗脚本。

# 实现说明
- Approach: 先补齐缺失 schema，把字段、必填项、权限、时间字段和唯一性目标写明；再在 `achievement-service` 中对齐“单用户-单卡片一条记录、单用户-单成就一条记录”的数据契约，并把索引要求明确到实现注释或交付说明中。
- Constraints: 必须保持现有云对象方法名和返回结构兼容；不得依赖删除线上历史数据来通过 schema 校验；若 uniCloud 仓库层无法声明索引，需要把控制台手工补索引作为显式交付步骤记录。
- Reuse: 继续复用现有 `users` / `cards` / `categories` / `favorites` schema 的字段命名与权限风格，复用 `achievement-service` 现有 `user_id`、`card_id`、`achievement_id` 命名，不引入新的业务字段缩写。

# 风险 / 回滚
- Risk: 新增 schema 与唯一性约束后，如果现有服务空间已经存在重复 `learning_log` 或重复 `user_achievements` 数据，部署阶段可能出现写入失败或需要人工清理；如果 `achievement-service` 仍保留“先查后写”而未处理并发冲突，则逻辑上仍可能打到唯一性错误分支。
- Rollback: 回退新增 schema 文件与 `achievement-service` 对齐逻辑；若服务空间已手工创建索引或唯一约束，需要同步在服务空间侧回退或放宽约束。

# 未决问题 / 假设
- 假设当前最需要收口的是 `learning_log` 与 `user_achievements`，而不是同步大规模重构 `users`、`cards`、`categories` 全部 schema。
- 假设现有服务空间允许后续补充 `user_id`、`(user_id, card_id)`、`(user_id, achievement_id)` 等索引；若仓库层不能声明索引，则通过部署说明补足。
- 假设当前用户接受“先把表设计合理，再进入下一轮查询优化”，即本轮以数据契约优先，不把所有性能问题一次性打包处理。

# 实施结果
- 已新增 `learning_log` 与 `user_achievements` schema，明确用户-卡片学习记录、用户-成就解锁记录的字段、权限和时间字段。

- 已调整 `achievement-service` 的写入策略，学习记录和成就记录改为优先使用确定性 `_id`，并兼容读取旧随机 `_id` 数据，降低并发下重复记录和重复发分的风险。
- 仓库层仍未发现可声明数据库索引的清单文件；服务空间中的联合索引和唯一性索引仍需后续在 uniCloud 控制台补齐。
