# 背景
当前仓库已经完成过一轮目录整理和基础复用抽象，但仍存在几类明显的高复杂度热点：前端页面 hook 里混合了展示计算、登录守卫、复制分享、任务状态同步和副作用编排；后台详情页和部分管理页仍保留重复的“确认弹窗 + loading + 请求 + 刷新”流程；云函数中的邀请奖励、积分增减与时间窗口判断逻辑也分散在多个服务里。  
这些代码虽然可运行，但理解成本高、跨层链路长，后续继续加任务渠道、积分规则或后台动作时，很容易在不同文件里复制同一套判断和文案，导致可读性、复用性和可扩展性继续下降。  
本次重构聚焦“拆职责、抽公共逻辑、缩短调用链路、补充必要中文注释”，不改变现有业务流程和接口形态。

# 目标
- 拆分前端长 hook 中的复杂职责，把邀请页、用户页和管理页详情中的重复流程提炼为更小的 helper/composable，降低单文件认知负担。
- 收敛云函数中的邀请绑定、积分日志、日期边界和管理员/用户重复更新逻辑，缩短链路并减少重复实现。
- 在复杂判断和关键业务规则处补充必要中文注释，提升后续维护者理解速度，同时保持行为兼容。

# 非目标
- 不调整页面路由、视觉样式和已有业务文案，只做等价重构或小范围的错误处理收敛。
- 不改动数据库 schema，不新增新的云服务，不重写整个 `src/utils/index.ts` 为大规模模块化工程。
- 不处理与本轮高复杂度热点无关的普通样式文件、静态资源或低收益格式化改动。

# 实现说明
- Approach: 先处理前端最高复杂度 hook，把纯计算、配置和副作用分层；再收敛后台详情页重复确认流；最后收敛云函数里的积分/邀请共性逻辑，并补充必要中文注释。
- Constraints: 必须保持现有页面入口、模板绑定字段、云函数方法名和返回结构兼容；新 helper 尽量放在现有目录下，不引入新的构建依赖或目录体系。
- Reuse: 继续复用 `usePageLayout`、`useLoginGuard`、`useConfirmedAction`、`getErrorMessage`、`showToast` 等已有基础能力，避免再次复制新的交互/鉴权封装。

# 风险 / 回滚
- Risk: 前端 hook 拆分后若返回字段或副作用时机处理不当，可能出现分享文案、登录刷新或后台操作反馈节奏变化；云函数抽公共逻辑时若积分余额和日志写入顺序处理不一致，可能引入奖励口径偏差。
- Rollback: 按层回滚本次新增 helper 与接入改动；优先回退页面 hook 和后台详情页，再回退云函数抽象层，保证可以分段止损。

# 未决问题 / 假设
- 假设用户这次“全局检查并优化”的重点是高复杂度、高链路和重复逻辑，而不是把所有文件统一做风格化重写。
- 假设仓库当前仍未配置 `lint` 与 `test:unit`，若命令不存在，将按 `skip + reason` 记录验证结果。

# 实施结果
- 已完成前端热点模块瘦身：邀请页与用户页分别新增 helper 文件，主 hook 只保留状态编排、副作用和用户交互，任务配置、分享文案、路由表、登录成功文案与只读映射逻辑统一下沉到纯函数层。
- 已完成后台重复链路收敛：`useConfirmedAction` 新增“普通异步反馈”能力，管理员用户详情页统一复用确认/执行/反馈模式，同时补上接口 `code !== 0` 时的失败断言，避免把业务失败误判为成功。
- 已完成云函数共享逻辑抽取：新增 `cloud-shared` common 模块，统一日期区间、积分流水、用户字段增量更新和分页回包；`user-center`、`points-service`、`admin-service` 复用后，邀请奖励与积分链路更短、更易扩展。
