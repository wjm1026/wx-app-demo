# 背景
当前前端页面目录以“功能域”分组，但每个目录内部同时平铺 `.vue`、`use*.ts` 和 `*.scss`，页面一多后可读性明显下降。  
尤其 `src/pages/admin`、`src/pages/user` 这类目录已经出现多组页面、hook、样式混放，查找入口文件、页面逻辑和样式文件都不够直观。  
本次调整聚焦“目录分层整理”，在不改页面路由、不改页面行为的前提下，把页面 hook 和样式下沉到统一子目录，降低后续维护成本。

# 目标
- 统一 `src/pages/**` 与 `src/components/CustomTabbar` 的目录规范，根目录保留 `.vue` 入口文件，新增 `hooks/` 与 `styles/` 子目录承载页面 hook 与样式文件。
- 修正所有受影响的相对导入与样式引用，确保 `pages.json`、页面跳转路径和现有业务行为保持不变。

# 非目标
- 不修改 `pages.json` 中的页面路由，不把 `.vue` 页面迁移到新的路由路径。
- 不调整页面模板结构、接口调用逻辑、权限逻辑或视觉样式设计，仅允许做等价路径修正。
- 不改动 `src/composables/**`、`src/api/**`、`uniCloud-aliyun/**` 等与本次目录整理无关的共享层与后端代码。

# 实现说明
- Approach: 统一采用“目录根保留页面/组件入口 `.vue`，逻辑下沉到 `hooks/`，样式下沉到 `styles/`”的结构；先移动文件，再批量修正 `script` 导入与 `style src` 路径，最后做类型校验。
- Constraints: 必须保持 `src/pages.json` 中的页面路径不变；不得修改页面行为、接口参数、样式选择器语义或组件对外契约；若某个目录只有单个页面，也沿用同一结构避免再次混乱。
- Reuse: 继续复用现有 `use*` 命名、`style src` 方式和 `@/` 别名，不额外引入 barrel、自动导出或新目录别名。

# 风险 / 回滚
- Risk: 文件移动后若有遗漏的相对路径，构建或类型检查会直接失败；若某些样式文件路径未同步，页面会出现样式丢失。由于本次触达多个页面目录和一个共享组件目录，风险高于单点整理。
- Rollback: 按目录回滚 `src/pages/*` 与 `src/components/CustomTabbar` 的文件移动和引用修改即可恢复；不涉及数据结构、接口契约和路由迁移，回滚边界清晰。

# 未决问题 / 假设
- 假设用户要求的“分一下类”指目录层级整理，而不是把页面路由路径也改成多级目录。
- 假设仓库当前没有单元测试文件；若验证时发现额外测试入口，需要一并修正路径。
