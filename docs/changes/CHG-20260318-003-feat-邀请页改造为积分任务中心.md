# 背景
当前 `pages/user/invite` 页面已经改成“赚取积分”任务中心，但邀请码补录能力仍停留在“登录前先缓存、登录时自动消费”这一条老链路上。  
用户进一步明确，希望把抖音、小红书等站外渠道也纳入真实可用的邀请闭环，让“已经登录但仍处于新用户阶段、尚未绑定邀请人”的用户，能够在赚积分页补填一次邀请码。  
这意味着本轮不再只是前端输入交互，而是要补齐一条服务端一次性绑定接口，并把“只能绑定一次、不能自邀、过期失效”等边界真正收拢到后端规则里。

# 目标
- 将邀请页改造成“赚取积分”任务中心，重构页面标题、Hero、任务列表、说明文案和统计摘要。
- 保留并复用现有邀请码分享链路，让“分享朋友”任务继续走真实分享路径。
- 为抖音、小红书、朋友圈等任务提供可执行的前端动作入口，例如复制传播文案、复制邀请码、朋友圈分享指引等。
- 为站外流量补充“手动输入好友邀请码”入口，并允许符合条件的新用户在登录后补绑一次邀请码。
- 在服务端落实邀请码补绑边界：只能使用一次、不能绑定自己、绑定成功后永久关闭、超出新用户时效后不可再绑。

# 非目标
- 不实现抖音、小红书、朋友圈任务的自动核销或自动发分。
- 不修改 `user-center` 现有邀请奖励规则和积分流水结构。
- 不引入新的页面路由或独立后台任务审核系统。
- 不支持无限期补绑邀请码；仅允许仍处于新用户窗口且尚未绑定邀请人的账号补填一次。

# 流程 / 行为摘要
- 用户进入原邀请入口后，首先看到的是“赚取积分”任务面板，而不是单一邀请码说明。
- “分享朋友”任务继续使用当前带邀请码的微信分享路径。
- “发朋友圈”任务通过页面级朋友圈分享配置和操作提示引导用户完成。
- “发抖音”“发小红书”等任务先提供文案/邀请码复制与发布引导，不承诺自动发分。
- 未登录用户仍可先保存好友邀请码，首次登录时继续自动消费。
- 已登录但尚未绑定邀请人、且仍处于新用户补绑窗口内的用户，会在赚积分页看到“手动输入好友邀请码”入口，并可提交绑定一次。
- 绑定成功后，页面立刻隐藏输入入口并刷新积分/邀请码状态；若邀请码无效、自邀、已过补绑窗口或已绑定过邀请人，则服务端直接拒绝。

# 实现说明
- Approach: 维持原路由 `/pages/user/invite` 不变，在同页内同时承载“分享任务中心”和“邀请码补绑入口”；前端继续复用分享 query 本地缓存能力，登录前仍可自动消费，登录后则走新增的 `bindInviteCode` 云对象接口进行一次性补绑。服务端根据用户 `inviter_id`、注册时间窗、邀请码合法性和自邀规则统一裁决是否允许绑定。
- Constraints: 现有分享链路不能回退；外部平台任务仍只做引导型前端交互；邀请码补绑必须以后端结果为准，前端只能做格式和状态提示；补绑入口不能对已经绑定邀请人或超出新用户窗口的账号开放。
- Reuse: 继续复用现有 `buildInviteSharePath`、`getStoredInviteCode`、`userApi.getInviteInfo()`、`onShareAppMessage` 以及微信登录时的自动邀请码消费规则。

# 风险 / 回滚
- Risk: 邀请码补绑从纯前端缓存升级为真实后端绑定后，若边界条件漏判，会出现重复奖励、自邀套利或老用户补绑等问题；同时前端如果仍按旧的“首次登录即锁死”逻辑判断，容易把可补绑的新用户错误挡掉。
- Rollback: 回退 `user-center` 的补绑接口与邀请信息扩展，并回退邀请页相关前端判断逻辑，即可恢复到“仅登录前自动消费邀请码”的旧实现。

# 未决问题 / 假设
- 假设本轮重点是把“邀请好友”升级为更完整的积分任务入口，而不是同时交付后台任务审核系统。
- 假设外部平台任务可接受“复制文案 + 复制邀请码 + 操作提示”的前端能力，后续如需自动核销再单独扩展服务端。
- 假设“登录后补绑一次邀请码”仍然受时间窗约束，本轮默认采用“注册后 24 小时内且未绑定邀请人”作为补绑窗口。

# 实施结果
- 已将原邀请页重构为“赚取积分”任务中心，页面包含积分总览、任务列表、邀请码素材卡和转化战绩区域。
