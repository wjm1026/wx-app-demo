# 背景
当前邀请页已经提供“复制邀请码”和“分享给微信好友”入口，但前端并没有把邀请码写进分享链接，也没有在好友打开小程序后自动落库到本地。  
后端 `loginByWeixin` 已支持接收 `inviteCode` 并在新用户注册时建立邀请关系，因此问题集中在前端链路未闭环。  
本次改动聚焦把“分享链接携带邀请码 -> 好友打开小程序自动缓存 -> 首次登录自动消费”这条链路接通。

# 目标
- 邀请页和用户中心的分享入口都能产出带 `inviteCode` 的小程序分享路径。
- 好友通过分享链接进入小程序后，前端自动缓存邀请码，登录时复用现有接口完成邀请关系绑定。

# 非目标
- 不新增手动输入邀请码页面或邀请码输入框。
- 不调整 `user-center` 云函数的邀请奖励规则和积分数值。
- 不新增分享海报生成功能或改动邀请页视觉布局。

# 流程 / 行为摘要
- 用户在邀请页或用户中心触发分享时，分享路径自动附带 `inviteCode`。
- 好友通过该链接打开小程序后，应用生命周期先把 `inviteCode` 存入本地 `INVITE_CODE`。
- 好友首次登录时，现有登录接口继续从本地读取 `INVITE_CODE` 并传给后端，后端按现有规则完成奖励发放。

# 实现说明
- Approach: 复用现有 `INVITE_CODE` 登录消费机制，不改接口签名；在分享出口补 path 参数，在应用入口补 query 落存，在登录页继续按已有逻辑读取和清理邀请码。
- Constraints: 分享路径必须继续使用现有页面 `/pages/user/invite`；不得引入新的状态管理层或改动云函数接口；保持未登录用户也能从分享链接进入页面。
- Reuse: 强制复用现有 `userApi.loginByWeixin(code, inviteCode)` 接口与 `INVITE_CODE` 本地存储约定。

# 风险 / 回滚
- Risk: 若分享 path、query 解析或本地缓存有误，会导致邀请码无法生效，或者旧邀请码被错误保留到后续登录；同时该链路跨越分享入口、应用生命周期与登录流程，属于跨页面行为改动。
- Rollback: 回退 `src/utils/index.ts`、`src/composables/useAppLifecycle.ts`、`src/pages/user/hooks/useInvitePage.ts`、`src/pages/user/hooks/useUserPage.ts` 即可恢复到当前无自动邀请码分享的状态。

# 未决问题 / 假设
- 假设当前主要投放渠道是微信好友分享，因此优先补 `onShareAppMessage`，不额外扩展朋友圈海报或其他渠道。
- 假设用户接受“分享链接自动携带邀请码”作为唯一邀请主路径，本轮不再补手动输入邀请码入口。
