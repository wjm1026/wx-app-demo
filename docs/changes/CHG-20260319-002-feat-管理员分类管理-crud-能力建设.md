# 背景
管理后台首页仍将“分类管理”标记为开发中，管理员无法直接维护分类名称、图标、封面和排序。  
当前虽然 `admin-service` 已有 `getCategoryList / saveCategory / deleteCategory` 接口，但缺少前端承接页面与上传交互，运营仍需手工改库。  
同时分类排序字段存在口径不一致风险：schema 使用 `sort`，后台实现仍包含 `sort_order`。  
本轮目标是补齐分类管理闭环，并统一排序字段口径，避免前后台展示顺序漂移。

# 目标
- 在管理员后台提供可用的分类管理页面，支持分类列表、新增、编辑、删除。
- 支持分类图标/封面本地选择上传并自动回填字段，降低手填 URL 成本。
- 统一分类排序字段口径（以 `sort` 为准）并保证管理端与用户端排序行为一致。

# 非目标
- 不实现分类批量导入/导出、拖拽排序、批量删除等高级运营能力。
- 不改动用户端分类页视觉结构，只保证其数据来源在本轮改动后保持兼容。
- 不实现“删除分类时自动迁移卡片到其他分类”；仍保留“有卡片则禁止删除”策略。

# 流程 / 行为摘要
- 管理员在后台首页点击“分类管理”进入分类列表，可查看分类基础信息与卡片数量。
- 管理员可新增或编辑分类，支持输入基础字段并在表单内上传图标/封面图片。
- 管理员删除分类时，服务端继续校验“分类下是否有卡片”，有卡片则拒绝删除并提示。
- 无流程图变化（本次无需 Mermaid）。

# 实现说明
- Approach: 复用卡片管理页的交互骨架，新增分类管理页面并采用“列表 + 弹层表单”模式。表单支持名称、图标、封面、渐变、排序、启用状态编辑；图片字段支持“上传回填 + 手输 URL”双模式。后端统一按 `sort` 字段排序与写入，并兼容历史 `sort_order` 数据读写。
- Constraints: 保持现有云对象方法名与返回结构不变（`getCategoryList/saveCategory/deleteCategory`）；删除分类时必须保留“有卡片不可删”约束；不改变用户端 `card-service.getCategories` 的入参与返回结构。
- Reuse: 复用 `adminApi`、`usePageLayout`、`useConfirmedAction`、`assertApiSuccess/showToast`，以及现有头像/卡片图片上传流程（`uni.chooseImage + uniCloud.uploadFile`）。

# 风险 / 回滚
- Risk: 排序字段从 `sort_order` 向 `sort` 收敛时，若兼容处理不完整，可能导致分类顺序异常；图片上传引入后，若云存储失败处理不足，可能出现保存时空 URL；删除校验若被误改，可能导致存在卡片的分类被删。
- Rollback: 回退新增分类页面与入口改动；恢复 `admin-service` 分类相关逻辑到变更前版本；后台首页分类入口重新回到“开发中”占位。

# 未决问题 / 假设
- 假设分类图标字段继续允许“emoji 或 URL”两种形式，本轮不强制限制格式。
- 假设分类数量规模较小，本轮列表页不做分页，直接全量加载。
- 假设历史数据可能存在 `sort_order`，实现时需要兼容读取并逐步回写到 `sort`。
