# 背景
当前“学习卡片解锁成就”链路已经会给用户加积分，但 `achievement-service` 只更新了 `users.points`，没有同步写入 `points_log`。  
结果是积分余额会增加，但积分明细页看不到对应的成就奖励流水，用户会误以为奖励没有到账或账务记录丢失。  
另外，`points_log` schema 当前也没有覆盖项目里已经在使用的 `invite_bonus` 类型；如果本轮新增 `achievement` 类型而不一起收口，积分流水类型契约会继续漂移。  
本次修复聚焦“成就奖励写流水并可展示”，不扩展到整套积分系统重构。

# 目标
- 在成就解锁时同步写入 `points_log`，让积分余额和积分流水保持一致。
- 为成就奖励流水引入明确的类型和幂等写入策略，避免重复补写。
- 让积分明细页对成就奖励使用明确的视觉标签，而不是落入泛化兜底样式。
- 让积分明细页在页面重新展示时先触发一次成就流水补写，再刷新流水列表，避免余额更新但列表仍停留在旧缓存。
- 顺手收口 `points_log` schema 中已实际使用但未声明的积分流水类型。

# 非目标
- 本次不重构 `points-service`、`user-center` 的整体积分发放架构。
- 本次不补做通用事务引擎，也不一次性解决所有历史流水缺失问题之外的账务一致性问题。
- 本次不改积分明细页布局，仅补充新流水类型的展示元数据。

# 流程 / 行为摘要
- 成就首次解锁时：创建 `user_achievements` 记录 -> 增加用户积分 -> 写入幂等的 `points_log` 成就奖励记录。
- 对已存在的成就解锁记录，如缺少对应成就流水，则只补写流水，不重复加积分。

# 实现说明
- Approach: 在 `achievement-service` 中引入 `points_log` 集合和现有 `cloud-shared` 积分辅助方法；成就首次解锁时按确定性 `_id` 写入 `points_log`，并在读取/校验成就时对历史已解锁但缺少流水的成就做幂等补写，只补流水不重复加分。前端积分明细页在 `onShow` 阶段会先调用 `achievementApi.getAchievements()` 触发补写，再同步刷新余额和流水列表，避免返回页面时看到旧分页缓存。
- Constraints: 必须保持 `recordLearning`、`getAchievements`、`checkAndUnlockAchievements` 返回结构兼容；不得对已到账积分再次加分；成就流水补写必须可重复执行且不产生重复记录。
- Reuse: 复用 `cloud-shared` 的 `appendPointsLog`、`incrementUserFieldsAndGetUser`，复用现有 `points_log` 字段命名与积分明细页兜底展示逻辑。

# 风险 / 回滚
- Risk: 若幂等流水 `_id` 规则设计不稳，会出现重复补写或误判已补写；若历史补写错误带上当前余额，可能让部分老流水的 `balance` 表意不准。
- Rollback: 回退 `achievement-service`、`points_log.schema.json` 和积分明细页映射改动；已补写的成就流水如需撤销，只能按 `_id` 规则在数据库中手工删除。

# 未决问题 / 假设
- 假设用户接受“历史缺失的成就流水只补记录、不回滚也不重算余额”。
- 假设成就奖励流水使用确定性 `_id` 是当前最小可行的幂等方案，不需要在本轮引入新的对账表。

# 实施结果
- 已在 `achievement-service` 中为成就奖励引入确定性积分流水 `_id`，首次解锁成就时会在加分后同步写入 `points_log`。
- 已在 `checkAchievements` 和 `getAchievements` 中补上历史缺失流水的幂等补写逻辑：若成就已解锁但没有对应 `achievement` 流水，则只补写流水，不重复加积分。
- 已补充积分明细页对 `achievement` 类型的专用视觉映射，并把页面回显时的刷新行为改为“先触发成就流水补写，再同步余额 + 流水列表”，避免出现余额已更新但列表仍停留在旧记录的状态。
- 已收口 `points_log` schema 的类型枚举，补齐 `achievement` 与项目内已使用的 `invite_bonus`。
