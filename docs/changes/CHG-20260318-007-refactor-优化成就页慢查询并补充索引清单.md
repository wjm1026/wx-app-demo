# 背景
当前成就页打开时，`getLearningProgress` 明显慢于 `getAchievements`。本地调试日志已经确认主要耗时集中在 `getCategoryProgress`，其内部对每个分类逐个查询 `cards`，形成典型 N+1 查询。  
上一轮已经完成 `learning_log`、`user_achievements` 的 schema 收口，这一轮在稳定数据契约的基础上，继续处理成就页最明显的慢查询，同时把需要在 uniCloud 服务空间手工补齐的索引清单明确记录，避免后续优化只停留在代码层。

# 目标
- 重构 `achievement-service` 中学习进度和成就统计的聚合逻辑，去掉按分类逐个查询卡片的慢路径。
- 在不改前端接口形态的前提下，把总卡片数、分类进度和已学卡片数收敛为更少的数据库查询。
- 明确 `learning_log`、`user_achievements`、`cards` 相关索引建议，方便后续在 uniCloud 服务空间中手工补齐。

# 非目标
- 本次不改页面模板、loading 交互和前端调用方式。
- 本次不继续扩展数据库 schema；上一轮 schema 收口已完成，本轮仅消费现有数据契约。
- 本次不引入新的云对象或新公共模块，不把成就系统整体重写成新的架构。

# 实现说明
- Approach: 以 `categories.card_count` 作为分类总卡片数和总卡片量的主来源；一次性读取用户学习记录，按已学 `card_id` 批量反查对应卡片的 `category_id`，在内存中聚合 `cardsLearned` 和 `categoryProgress`；`checkAchievements` 与 `getLearningProgress` 复用同一套聚合逻辑，避免重复慢查询。
- Constraints: 必须保持 `getLearningProgress`、`getAchievements`、`checkAndUnlockAchievements` 的返回结构兼容；必须兼容上一轮遗留的旧学习记录数据；索引只能提供清单，不能假装已在服务空间自动创建。
- Reuse: 复用上一轮新增的确定性学习记录/成就记录契约与现有 `categories.card_count` 字段，不新增新的缓存表。

# 风险 / 回滚
- Risk: 若 `categories.card_count` 与真实启用卡片数存在历史漂移，重构后总量展示可能与旧逻辑不同；若批量按 `card_id` 查询的分片策略处理不当，可能在已学卡片很多时引入遗漏或重复统计。
- Rollback: 回退 `achievement-service/index.obj.js` 到当前实现即可恢复原查询路径；索引清单本身不影响运行时。

# 未决问题 / 假设
- 假设 `categories.card_count` 已由管理端新增/删除卡片时维护，可作为当前分类总卡片数的可信来源。
- 假设 uniCloud 的 `db.command.in` 可用于按已学卡片 ID 批量读取卡片分类信息；若存在单次数量限制，则通过分片查询规避。
- 假设用户已明确同意开始这一轮实现，无需再次等待额外确认。

# 实施结果
- 已将成就页学习进度聚合改为“分类一次查询 + 学习记录一次查询 + 已学卡片按 `card_id` 分片回查分类”的模式，去掉原先按分类逐个查询卡片的 N+1 慢路径。
- `getLearningProgress` 与 `checkAchievements` 现在共用同一套学习进度聚合逻辑，避免在同一请求链路里重复统计学习记录和分类完成度。
- 总卡片数和分类总量改为优先使用 `categories.card_count`，已学卡片数改为基于有效启用卡片去重统计，兼容旧重复学习记录数据。
- 已补充需在 uniCloud 服务空间手工创建的索引清单：
  - `learning_log(user_id)`
  - `learning_log(user_id, card_id)`
  - `user_achievements(user_id)`
  - `cards(category_id, status)`
  - `cards(status)`（优先级次于前四项）
