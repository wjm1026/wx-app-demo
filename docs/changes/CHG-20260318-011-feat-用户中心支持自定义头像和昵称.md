# 背景
用户中心当前只展示头像和昵称，其中头像点击后仍提示“头像功能开发中”，昵称也没有任何编辑入口。  
后端已有 `updateUserInfo` 云对象可写昵称和头像，因此问题集中在前端链路未闭环，导致登录后的用户无法自定义个人资料。

# 目标
- 用户可在用户中心修改自己的头像，并将结果持久化到现有用户资料。
- 用户可在用户中心修改自己的昵称，并在更新后立即看到最新展示。

# 非目标
- 不调整登录方式，不改动登录成功后的用户建档逻辑。
- 不改动后台用户管理页面和邀请页的资料编辑入口。
- 不新增数据库字段或改动现有用户表结构。

# 流程 / 行为摘要
- 微信小程序端已登录用户修改头像时，使用官方 `button open-type="chooseAvatar"` 获取头像临时路径，前端上传到 uniCloud 后调用现有 `updateUserInfo` 写回头像地址。
- 微信小程序端已登录用户修改昵称时，使用官方 `input type="nickname"` 作为资料输入控件，在失焦或确认时校验并调用现有 `updateUserInfo` 写回昵称。
- 非微信端仍保留现有相册选择和通用输入框兜底路径。
- 未登录用户触发上述入口时，沿用现有登录流程。

# 实现说明
- Approach: 复用现有 `userApi.updateUserInfo`，微信端头像改为官方 `chooseAvatar` 控件获取临时路径后上传，昵称改为官方 `input type="nickname"` 失焦/确认时保存；非微信端继续复用现有 `uni.chooseImage + uni.showModal` 回退方案。
- Constraints: 仅修改前端 `src/pages/user/**` 和本变更文档；保持现有 store、登录守卫与接口名称不变；兼容当前阿里云 `uniCloud-aliyun` 服务空间。
- Reuse: 复用 `showToast`、`showLoading`、`hideLoading`、`getErrorMessage`、`loadUserInfo`、`store.setUserInfo` 现有能力，不新增状态管理层。

# 风险 / 回滚
- Risk: 头像上传依赖 uniCloud 前端直传，若云存储配置异常会导致头像更新失败；微信端昵称输入改为官方控件后，需注意失焦和确认事件可能重复触发。
- Rollback: 回退 `src/pages/user/**` 对应改动，恢复头像点击仅触发旧占位逻辑。

# 未决问题 / 假设
- 假设当前阿里云服务空间返回的 `fileID` 可直接作为头像 URL 使用，这与官方文档针对阿里云的说明一致。
- 假设当前项目微信基础库已支持 `chooseAvatar` 与 `input type="nickname"`，与仓库依赖类型定义保持一致。
