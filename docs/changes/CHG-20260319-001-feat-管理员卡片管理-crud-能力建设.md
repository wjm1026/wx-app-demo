# 背景
当前管理后台首页已展示“卡片管理”入口，但该模块仍是 `开发中` 状态，管理员无法在后台维护卡片内容。  
与此同时，云端 `admin-service` 已具备 `getCardList / saveCard / deleteCard` 基础能力，前端缺少对应页面与交互流程。  
如果继续依赖手工改库或脚本维护卡片，运营调整内容效率低且容易引入字段错误。  
本轮目标是补齐管理员卡片管理 CRUD 流程，并同时收敛一处已知数据一致性风险。

# 目标
- 在管理员后台提供可用的卡片管理页面，支持列表查询、新增、编辑、删除四个核心动作。
- 复用现有后台风格与分页模式，保证管理员模块交互一致性。
- 修复卡片更新时跨分类移动导致 `categories.card_count` 可能失真的风险。
- 支持管理员在卡片编辑表单内直接选择本地图片并上传，自动回填 `image` 字段。

# 非目标
- 不实现音频/视频资源上传，本轮仅补齐卡片主图上传能力。
- 不新增“分类管理”页面或分类体系重构，仅调用现有分类列表供卡片选择。
- 不改动前台卡片详情、搜索、推荐算法及用户端展示逻辑。
- 不引入批量导入/导出、批量编辑等高级运营能力。

# 流程 / 行为摘要
- 管理员在后台首页点击“卡片管理”进入列表页，按关键词/分类/状态筛选并分页查看卡片。
- 管理员可在列表中发起“新增”或“编辑”，提交后调用 `saveCard` 持久化并刷新列表。
- 管理员可删除卡片，执行确认后调用 `deleteCard`，并同步刷新当前列表状态。
- 无流程图变化（本次无需 Mermaid）。

# 实现说明
- Approach: 采用“单页面 + 内嵌编辑态”方案实现卡片 CRUD。列表层复用 `usePagedList`，筛选条件为 `keyword/categoryId/status`；编辑态复用 `saveCard` 完成新增与更新；删除走 `useConfirmedAction`。后端在 `saveCard` 更新分支补充“旧分类 -1 / 新分类 +1”计数修正。
- Constraints: 保持现有云对象方法名与响应结构不变；字段命名沿用 `cards` schema（如 `category_id`、`is_free`、`points_cost`）；仅在当前目录分层规范（`hooks/`、`styles/`）内新增文件，不做目录重排。
- Reuse: 复用 `adminApi` 云对象调用模式、`usePagedList` 分页封装、`usePageLayout` 顶部布局、`useConfirmedAction` 确认执行流、`assertApiSuccess/showToast/navigateTo` 工具链。

# 风险 / 回滚
- Risk: 卡片更新时若分类计数修正逻辑处理不完整，可能导致 `categories.card_count` 与真实卡片数继续漂移；管理端新增表单字段若与现有 `cards` 数据不兼容，可能导致历史卡片编辑失败；删除仍为硬删除，误操作影响不可逆。
- Rollback: 回退本轮新增后台页面与入口改动；将 `admin-service` 的 `saveCard` 逻辑恢复到变更前版本；前端后台入口重新标记为 `开发中`，维持现状流程。

# 未决问题 / 假设
- 假设本轮“删除”维持现有硬删除语义，不额外改为软删除。
- 假设卡片主图/音频/视频资源由外部已可访问 URL 提供，本轮不接入上传能力。
- 假设管理员卡片编辑字段以当前 `cards.schema` 为准，不新增 schema 字段。

# 实施结果
- 已新增管理员卡片管理页面，支持列表查询、按状态/分类/关键词筛选、分页加载、新增/编辑表单保存和删除确认流程。
- 已打通后台首页“卡片管理”入口，从占位态改为可直接跳转到新页面。
- 已在 `admin-service.saveCard` 更新分支补齐跨分类迁移时的 `categories.card_count` 增减修正，降低分类卡片统计漂移风险。
- 已在 `card-service` 分类查询接口增加“按启用卡片实时聚合计数”兜底，避免历史脏数据导致前台分类卡片数显示不准确。
- 已支持管理员在卡片编辑弹层里直接选择本地图片并上传到云存储，上传成功后自动回填主图 URL。
