# 背景
当前首页左上角 Logo 与游戏页卡片信息主要写死在前端，运营无法在 admin 端动态维护。  
需求要求将“首页 Logo + 游戏列表（含封面图、图标、启动模式、跳转路径、可玩状态）”统一纳入管理后台，并保持现有“听音选图先选分类”流程不变。  
同时后端与前端都需按现有分层和门禁规范落地，并更新 OpenAPI 与变更记录。

# 目标
- 新增后端展示配置数据模型与接口：管理员可读写，前台可读取启用项。
- 管理端新增“游戏配置”页面，支持首页 Logo、游戏列表的新增/编辑/删除与 icon/cover 上传。
- 用户端首页与游戏页改为配置驱动，并保持 listen-pick 分类弹层逻辑。

# 非目标
- 不扩展品牌文案等额外字段，仅配置首页 Logo 图标。
- 不新增匿名放行策略，前台接口继续沿用现有鉴权策略。
- 不改造既有 tabbar 与全局视觉风格，仅接入配置数据。

# 流程 / 行为摘要
- 管理员进入 admin 首页，点击“游戏配置”进入维护页。
- 顶部维护首页 Logo（支持上传与手动输入），下方维护游戏项（新增/编辑/删除/排序/上传 icon+cover）。
- 保存时整表提交：后端执行新增、更新、删除同步，并做 key/路径/枚举等校验。
- 用户端读取 `/api/v1/display-config`：首页展示配置 logo，游戏页按 `enabled + sortOrder` 渲染并按 `launchMode + routePath + available` 执行点击行为。

# 实现说明
- Approach: 后端新增 `mini_app_configs + game_configs`，并通过 `DisplayConfigService` 提供 admin/user 两类读取与 admin 保存；保存采用“整表同步”策略（新增/更新/删除）。前端管理端新增独立配置页，上传复用 `/api/v1/files/upload`，保存前做 key 唯一、必填、路径与枚举校验。用户端首页/游戏页接入 display-config 接口，保持听音选图分类弹层行为。
- Constraints: 严格沿用现有分层（Controller -> AdminService -> Ops/Service -> Mapper）与异常规范（`BusinessException`）；`routePath` 必须 `/pages/` 开头；`launchMode` 仅 `listen-pick-category|direct`；`tone` 仅现有六档。
- Reuse: 复用现有 `ResponseMapper`、`ApiResponseMediaSignAdvice`、`uploadApiFile`、页面 hook 组织方式与 admin 页面结构。

# 风险 / 回滚
- Risk: 整表同步若前端提交不完整会删除未提交项；路径配置错误会造成跳转失败；历史游戏 key 变更会影响运营配置稳定性。
- Rollback: 回退 `V3` 相关读写代码与前端 display-config 页面入口；用户端恢复静态游戏卡片和静态 logo。

# 未决问题 / 假设
- 假设 `game_key` 一旦上线后由运营稳定维护，不在运行期频繁改 key。
- 假设游戏封面图尺寸与裁切策略由前端现有卡片布局承接，本轮不新增图片处理参数。
- 假设仓库当前存在历史门禁债务（超长文件）由后续专项治理，不在本次需求内重构。
