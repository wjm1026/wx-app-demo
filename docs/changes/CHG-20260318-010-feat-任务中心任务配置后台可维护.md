# 背景
当前“赚取积分”页的任务卡片仍由前端硬编码，任务名称、描述、奖励积分、按钮文案和排序都散落在 `invite-page.helpers.ts` 里。  
这导致每次运营想调整任务包装文案或奖励额度时，都必须重新发版，管理员后台也无法维护这块配置。  
本轮目标是把任务卡片的展示字段收敛到云端配置，并在管理员后台提供一个稳定的维护入口，同时保留现有任务键对应的真实行为逻辑。

# 目标
- 将积分任务的展示字段改为云端可配置，包括任务名称、渠道名、描述、补充说明、奖励积分、按钮文案、排序和启用状态。
- 在管理员后台新增“任务配置”页面，让管理员可以查看、编辑并保存这组任务配置。
- 前台邀请/任务中心页面改为读取云端任务配置渲染，未配置时自动回退到默认配置。

# 非目标
- 不实现任务自动审核、自动核销或按配置动态执行不同业务动作。
- 不新增新的任务奖励发放规则；本轮只配置展示字段，不改变现有分享/补绑邀请码逻辑。
- 不把任务图标、任务动作类型或分享实现改成完全可视化编排，仍保留“固定任务键映射固定行为”的约束。

# 流程 / 行为摘要
- 用户进入“赚取积分”页时，页面先从云端拉取任务配置，再结合本地登录态和邀请码状态渲染任务卡片。
- 管理员进入后台后，可从首页模块入口进入“任务配置”页，对固定任务键的展示字段做批量编辑并保存。
- 若数据库尚无任务配置记录，前后台都应自动回退到默认任务配置，避免页面空白或后台无法初始化。

# 实现说明
- Approach: 以固定任务键（如 `share-friend`、`moments`）作为行为主键，云端只保存可配置展示字段；前台根据任务键拼接静态图标/行为，后台通过批量保存接口维护整组配置。若数据库无记录，则服务端返回默认配置并由后台可直接编辑后落库。
- Constraints: 不改现有邀请分享、邀请码补绑和积分流水逻辑；管理员配置必须保留固定任务键集合，避免出现未知任务键破坏前台行为；前台未登录态也需要拿到任务配置，因此读取接口需允许匿名访问。
- Reuse: 复用现有 `userApi` / `adminApi` 云对象调用方式、`usePageLayout`、`assertApiSuccess`、`showToast`、后台首页模块卡片模式，以及邀请页现有任务动作处理函数。

# 风险 / 回滚
- Risk: 云端配置一旦缺字段、顺序错乱或引入未知任务键，会直接影响任务页渲染和后台维护；若公开读取接口权限处理不当，还可能让前台拿不到配置数据。
- Rollback: 回退任务配置集合和前后台新增接口，前端恢复到 `invite-page.helpers.ts` 的硬编码默认任务卡片即可。

# 未决问题 / 假设
- 假设本轮“可配置项”限定为展示层字段，不把任务实际动作类型、图标资源和分享执行代码做成后台可编辑。
- 假设管理员维护的是一组固定任务，而不是随意新增/删除任务键；因此后台提供的是编辑与启停，而不是自由创建任意任务。
